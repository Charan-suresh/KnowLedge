📋 System Architecture Overview

What is KnowLedge?

KnowLedge is a Comprehension Debt Tracker (CDT) that addresses AI-induced over-reliance in education by tracking, measuring, and systematically clearing "comprehension debt"—concepts students passively copy without truly understanding.

Core Philosophy

Instead of instantly answering questions (traditional AI tutoring), KnowLedge:

  • Detects borrowed concepts automatically (Scout)
  • Tracks learning debt across a personal ledger
  • Guides students through Socratic clearing (Sage)
  • Verifies true mastery via multimodal assessment (Lens)

4. Orchestrator (Workflow Coordinator)

Role: Manages concurrent operations of Scout, Sage, and Lens with resource constraints.

Background Processing

Scout runs in daemon thread while main app serves requests

Resource Management

Mutex lock prevents simultaneous Sage + Lens execution in LOW_RAM mode

Event Bus

Broadcasts events (DEBT_ADDED, CLEARING_COMPLETE) for UI updates

5. RAG (Retrieval-Augmented Generation)

Role: Grounds Sage dialogue in syllabus content to prevent hallucinations.

Syllabus PDF ↓ pdfplumber (extract text) ↓ Text chunking (500 chars, 100 char overlap) ↓ ChromaDB (vectorize + store) ↓ Query: retrieve top-3 chunks semantically similar to concept ↓ Inject into Sage system prompt

🔄 Data Flow

Concept Tracking (Scout)

Student Input ↓ POST /api/tag (Scout.tag_content) ↓ Gemma-4-E2B (Function Calling) ↓ [{concept_tag, confidence}] ↓ db.insert_debt() → status='on_loan' ↓ Event: DEBT_ADDED ↓ Dashboard reflects new concept

Clearing Session (Sage)

User clicks concept → Sage modal opens ↓ POST /api/sage (concept, chat_history) ↓ RAG.retrieve_context(concept) ↓ Gemma-4-E4B (Socratic Dialogue) ↓ Student explains concept ↓ LLM Decision: ├─ Correct & Complete → verify_comprehension() ├─ Incomplete → Ask probe └─ Wrong → Point out gap ↓ IF cleared: db.update_debt_status("clear") ↓ Dashboard updates badge

Vision Verification (Lens)

User uploads handwritten image ↓ POST /api/lens (image, concept) ↓ Image validation & encoding ↓ Gemma-4-E4B Vision ↓ Output: {x, y, width, height, explanation} ↓ IF gaps found: db.update_debt_status("persists") ↓ Frontend renders feedback overlay with bounding box

💾 Database Schema

debt_log (Concept Tracking)

CREATE TABLE debt_log (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    concept TEXT NOT NULL,
    source_text TEXT,
    timestamp TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    confidence REAL,           -- 0.0-1.0 (Scout confidence)
    status TEXT DEFAULT 'on_loan'  -- 'on_loan'|'clear'|'persists'
);

CREATE INDEX idx_concept ON debt_log(concept);
CREATE INDEX idx_status ON debt_log(status);

clearing_history (Session Logs)

CREATE TABLE clearing_history (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    concept TEXT NOT NULL,
    session_ts TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    result TEXT,              -- 'cleared'|'not_cleared'
    notes TEXT                -- Sage feedback or Lens explanation
);

Common Queries

-- Get all On Loan concepts
SELECT concept, confidence, timestamp
FROM debt_log
WHERE status = 'on_loan'
ORDER BY timestamp DESC;

-- Weekly progress
SELECT DATE(timestamp) as date,
       SUM(CASE WHEN status = 'clear' THEN 1 ELSE 0 END) as cleared,
       COUNT(*) as total
FROM debt_log
GROUP BY DATE(timestamp);

🚀 Deployment & Configuration

Local Development Setup

# 1. Clone and setup
git clone https://github.com/Charan-suresh/Gemma4-CDT.git
cd Gemma4-CDT
python3 -m venv venv
source venv/bin/activate

# 2. Install dependencies
pip install -r requirements.txt

# 3. Start Ollama (Terminal 1)
ollama serve

# 4. Pull models (Terminal 2)
ollama pull gemma:4-e2b
ollama pull gemma:4-e4b

# 5. Run application (Terminal 3)
python3 -m uvicorn knowledge.main:app --reload --port 8000

Production Deployment

Recommended Stack:
  • ASGI Server: Gunicorn with Uvicorn workers
  • Reverse Proxy: Nginx
  • TLS: Let's Encrypt
  • Database: SQLite or PostgreSQL
  • Container: Docker (optional)

Performance Optimization

  • Background Processing: Scout runs in daemon thread, doesn't block requests
  • Context Caching: RAG results cached with @lru_cache
  • Low RAM Mode: Mutex lock prevents simultaneous heavy model execution
  • Image Compression: Resize handwritten solutions before vision processing
